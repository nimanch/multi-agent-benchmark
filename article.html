Evaluating 9 AI Agent Combinations on the Same Snake Game

Nishant Manchanda · March 2026

There's a new AI coding agent every week. Most benchmarks compare them on vibes. I wanted something more controlled: give multiple agent combinations the same spec, same language, same machine, and compare the output code systematically.

I picked three multi-agent orchestrators (Superpowers, DeerFlow 2.0, Squad) and three spec-driven toolkits (GSD, Spec Kit, OpenSpec), producing a 3×3 matrix of nine implementations. The task: a terminal Snake game in Python curses, from a shared spec.

Then I had an LLM judge score each implementation across five dimensions.

The results surprised me. The “best architecture” had a bug. The simplest approach swept the top three spots. And the spec toolkit shaped the code structure more than the orchestrator did.

The Tools

Before getting into results, here's what each tool actually does. These aren't interchangeable — they solve fundamentally different problems.

Orchestrators

Superpowers (94K+ stars) is a subagent-driven development framework. When you give it a task, it doesn't just generate code — it dispatches specialized subagents in parallel. A planning agent breaks the problem down. Implementation agents write code concurrently. Then a two-stage review process kicks in: first a spec-compliance check (does the code match the requirements?), then a code-quality review (is the code well-structured?). Superpowers uses a skill-based architecture where agents have defined capabilities and constraints. It ran natively in this experiment — the only orchestrator that did.

DeerFlow 2.0 (ByteDance) follows a Research → Plan → Code → Review pipeline. What distinguishes it is the research phase: before any code is written, a research agent examines best practices, common patterns, and potential pitfalls for the task at hand. DeerFlow also maintains sub-agent memory across tasks within a session. It requires Python 3.12+ and couldn't run natively on our ARM64 test machine. Its workflow was simulated following the documented architecture.

Squad (v0.8.25) is a fundamentally different kind of tool. It's a team management layer for GitHub Copilot that creates persistent specialist roles — frontend developer, backend developer, tester, team lead — stored in .squad/ directories within your project. Squad is designed for ongoing GitHub Issues and PR workflows, not single-prompt generation. In this experiment, the Squad runs were effectively single-agent Copilot sessions with access to Squad's context files. No true multi-agent orchestration occurred.

Spec Toolkits

GSD (Get Shit Done) (35K+ stars) takes a milestone/phase approach. Philosophy: ship fast with accountability. You run new-project to scaffold milestones, execute-phase to work through each phase with fresh context windows, then verify-phase to check output. Atomic git commits per phase create a clear audit trail.

Spec Kit (72.7K stars) is GitHub's approach. It generates feature scenarios (given/when/then), acceptance criteria, and architectural specifications — structured documents designed to be consumed by AI agents. In practice, this led agents to build more elaborate class hierarchies and typed interfaces.

OpenSpec is the most lightweight: plan → apply → archive. Minimal ceremony. Task breakdowns in plan.md files give agents freedom in code organization, which showed in the most diverse architectures across orchestrators.

Methodology

What was controlled:

  • Identical spec (SNAKE_SPEC.md) — 16 acceptance criteria
  • Same language: Python 3.x with curses (stdlib only)
  • Same machine: NVIDIA Jetson ARM64, 8GB RAM, Ubuntu
  • Single-file implementations, zero manual interventions

What was NOT controlled — and this matters:

  • Superpowers ran natively as a fully multi-agent system. The only orchestrator that operated as designed.
  • DeerFlow 2.0 couldn't run natively (Python 3.12+ required). Its pipeline was simulated manually.
  • Squad was installed (v0.8.25), squad init ran, but --agent squad doesn't register as a Copilot agent. The experiments used single-agent Copilot with Squad's .squad/ context files. Squad is designed for persistent team roles across PR workflows, not single-prompt generation.

Evaluation: An LLM judge scored each implementation 1–5 on five dimensions: Spec Compliance, Correctness, Code Quality, Completeness, and Robustness.

Two things jump out:

  1. Only 4 of 9 implementations checked minimum terminal size — three GSD variants plus squad-openspec. This is a clear spec requirement that 5 implementations simply ignored.
  2. Two implementations (superpowers-speckit, squad-speckit) have a grow-timing bug where the snake doesn't visually grow until one frame after eating.

The Grow-Timing Bug

The spec says “the snake grows by one segment each time it eats food.” Two implementations get this subtly wrong due to an architectural choice.

Here's the pattern that causes the bug — from superpowers-speckit:

# Game loop (simplified):
snake.move()                    # advance first
if snake.head == food.position:
    snake.grow()                # sets grow_pending = True
    food.respawn(...)

# Inside Snake.move():
def move(self):
    new_head = (self.head[0] + dy, self.head[1] + dx)
    self.body.insert(0, new_head)
    if not self.grow_pending:   # checked BEFORE grow() is called
        self.body.pop()
    else:
        self.grow_pending = False

Compare with the correct pattern used by 7 of 9 implementations:

def move_snake(snake, direction, food):
    new_head = (head_y + dy, head_x + dx)
    new_snake = [new_head] + snake[:]
    ate = new_head == food
    if not ate:
        new_snake.pop()         # grow decision is immediate
    return new_snake, ate

The implementation with the best architecture score (superpowers-speckit: Code Quality 5/5, five clean OOP classes) had a functional bug. The procedural implementations with no classes got the behavior right.

The Dead EventBus

deerflow-openspec built an event-driven architecture with an EventBus class. Events are emitted throughout the game — EVT_MOVE, EVT_EAT, EVT_DIE. But bus.on() is never called anywhere. Zero subscribers. The events fire into the void.

The game works fine — the EventBus just doesn't do anything. It's the code equivalent of installing a fire alarm system and never connecting the sirens.

One redeeming quality: deerflow-openspec is the only implementation that uses collections.deque for the snake body, giving O(1) head insertion vs O(n) for list insert(0, ...).

What the Spec Toolkit Controls vs. What the Orchestrator Controls

The spec toolkit shaped architecture:

  • GSD → procedural, pragmatic, “just make it work”
  • Spec Kit → OOP, typed, separated concerns, elaborate class hierarchies
  • OpenSpec → functional, state-as-data, testable pure functions

The orchestrator shaped completeness and robustness:

  • All three GSD variants checked terminal size. Of non-GSD, only squad-openspec did.
  • deerflow-gsd was the only implementation with a food-spawn safety net (1000 attempts with fallback)
  • Superpowers consistently produced more polished rendering (manual Unicode box-drawing borders)

GSD's milestone-based approach forced all three orchestrators to handle edge cases that other toolkits didn't prompt for. The spec says “Minimum terminal size: 20x10” — GSD treated this as a requirement; Spec Kit and OpenSpec (except squad-openspec) didn't surface it as a first-class criterion.

Limitations

Snake is too easy. All nine implementations work. The interesting differences are in edge cases and code style — not in whether the agents could solve the problem.

Only one orchestrator ran natively. DeerFlow was simulated (Python 3.12+ requirement). Squad was installed but doesn't function as a multi-agent system for single-prompt generation — the experiments used single-agent Copilot with Squad's context files. Only Superpowers operated as true multi-agent orchestration.

N=1 per combination. Each combination ran once. LLM outputs are stochastic. Running each five times and averaging would be more rigorous.

LLM-as-judge. The evaluation was done by an LLM, not a test suite. The grow-timing bug was caught by code inspection, not automated testing.

Scoring granularity. The actual spread was 17–22, a 5-point range across 9 implementations. The top-4 vs bottom-5 distinction is clearer than individual rankings.

Conclusions

Three data-driven takeaways:

  1. GSD's spec compliance advantage was decisive. All three GSD variants scored 5/5 on spec compliance and checked terminal size. deerflow-gsd led at 22/25; squad-gsd and superpowers-gsd scored 21 and 20. The toolkit's milestone-based approach — fresh context per phase, explicit verification — enforced spec requirements more reliably than scenario-based (Spec Kit) or task-based (OpenSpec) approaches.
  2. Architectural ambition correlated negatively with correctness. The two implementations with the most classes (superpowers-speckit, squad-speckit) both had the grow-timing bug (Correctness: 3/5 each). The EventBus implementation had dead code (Code Quality: 3/5). The top four scorers (22, 21, 21, 20) were all procedural or simple functional code.
  3. The spec toolkit shaped the code more than the orchestrator. Across all three orchestrators, GSD produced procedural code, Spec Kit produced OOP, OpenSpec produced functional. The orchestrator affected quality within those patterns but didn't change the patterns themselves.

Whether this generalizes beyond Snake — beyond a well-understood, small, single-file problem — is an open question.
